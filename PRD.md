# Along - 多 Agent 智能助手

## Product Requirement Document (PRD) V1.0

------

# 一、产品定位

## 产品愿景

构建一个长期存在于用户电脑中的多 Agent 智能助手。

它不是：

- AI 聊天机器人
- AI 客服
- AI 搜索工具

而是：

```text
面向开发者的个人 AI 助手
通过多个专业化 Agent 协作
提供对话、规划、记忆、反思等能力
```

在用户的：

- 学习
- 工作
- 项目开发
- 规划与反思

过程中持续提供支持。

------

## 核心理念

### Multi-Agent First

多 Agent 协作优先。

而非单一工具。

任何能力最终都通过专业化 Agent 协作输出。

------

### Memory Driven

记忆驱动。

记录：

```text
用户偏好
项目进展
学习轨迹
关键事实
```

而非仅记录聊天记录。

------

### One Identity

用户永远只面对一个助手：Along。

即使内部存在多个 Agent。

用户感知层永远只有统一的专业形象。

------

```yaml
产品名: Along
```

## 2.2 核心能力

- 记忆系统：长期记忆用户偏好、项目信息、关键事实
- 信息整合：搜索、调研、内容归纳
- 反思复盘：周期性回顾成长与进展
- 工具调用：文件系统、Git、浏览器等

# 三、系统架构

## 3.1 用户可见层

仅存在：

```text
Along（多 Agent 智能助手）
```

------

## 3.2 内部 Agent 架构

```text
Companion Core
│
├── Emotion Agent      # 情绪识别与状态支持
├── Planner Agent      # 学习与项目规划
├── Research Agent     # 搜索与调研
├── Tool Agent         # 工具调用
├── Memory Agent       # 记忆管理
├── Reflection Agent   # 复盘分析
├── Web Agent          # 网页搜索
├── Summarize Agent    # 信息整合
└── File Generation Agent # 文档生成
```

------

## 3.3 路由机制

Companion Core 使用**规则引擎 + 关键词匹配**的方式决定调用哪个 Agent：

| 规则 | 目标 Agent | 示例关键词 |
|------|-----------|------------|
| 包含情绪词 | Emotion Agent | "烦"、"累"、"焦虑"、"难过"、"难受" |
| 包含学习/规划词 | Planner Agent | "学习"、"规划"、"目标"、"计划"、"安排" |
| 包含搜索词 | Research Agent | "搜索"、"调研"、"分析"、"了解"、"查一下" |
| 包含记忆词 | Memory Agent | "记得"、"回忆"、"忘记"、"之前说过" |
| 包含复盘词 | Reflection Agent | "复盘"、"总结"、"回顾"、"这段时间" |
| 包含工具词 | Tool Agent | "读取文件"、"git status"、"打开链接" |

**兜底机制：** 当无法匹配任何规则时，由 Companion Core 直接响应，并在响应后将对话内容写入 Memory Agent。

# 四、Agent 设计

## 4.1 Companion Core

系统核心。

**职责：**
- 统一交互入口（Along）
- 决策路由（规则引擎 + 关键词匹配）
- 输出控制
- 上下文维护

所有 Agent 输出必须经过 Companion Core。

------

## 4.2 Emotion Agent

情绪识别与状态支持。

**职责：**
- 情绪识别：焦虑、沮丧、疲惫、兴奋、迷茫
- 支持策略：倾听、理解、建议、客观分析

**触发场景：**
- 用户表达负面情绪
- 用户语气异常
- 默认兜底 Agent

------

## 4.3 Planner Agent

学习与项目规划。

**职责：**
- 学习规划：Go、AI Agent、英语等
- 项目规划：个人产品、技术项目等
- 长期目标拆解：年目标 → 月目标 → 周目标

------

## 4.4 Research Agent

搜索与调研。

**职责：**
- 搜索
- 技术调研
- 产品调研
- 竞品分析

------

## 4.5 Tool Agent

工具调用管理。

**V1 支持：**
- 文件系统（读取/写入）
- Git（查看提交记录、仓库状态）
- 浏览器（打开链接）

**未来支持：**
- 邮件
- 日历
- MCP

------

## 4.6 Memory Agent

记忆管理，产品核心。

详见「七、记忆系统」。

------

## 4.7 Reflection Agent

用户主动触发的复盘工具。

**职责：**
- 用户成长分析
- 项目复盘
- 记忆压缩
- 生成观察

**触发方式：**
- 用户在"我们"页面点击"复盘"按钮
- 用户在对话中明确要求"复盘"、"总结"、"回顾"

------

## 4.8 Web Agent

网页搜索与信息获取。

**职责：**
- 联网搜索
- 网页内容获取
- 实时信息查询（天气、汇率等）

------

## 4.9 Summarize Agent

信息整合。

**职责：**
- 对原始信息进行去重、分类、提炼
- 输出结构化摘要
- 支持简报、详报、技术报告三种类型

------

## 4.10 File Generation Agent

文档生成。

**职责：**
- 按标准模板生成 Markdown 文档
- 支持研究报告、周报、会议纪要等模板
- 文档自动保存到本地

# 六、记忆系统

## 6.1 记忆存储方案

采用**SQLite + 本地 JSON 文件**组合方案：

- SQLite：存储结构化记忆（事实、偏好、关系等）
- JSON 文件：存储原始对话记录和非结构化回忆
- 向量数据库：V1 暂不引入，使用关键词索引进行检索

所有数据存储于用户本地，不进行云同步。

------

## 6.2 记忆分层

### L1：事实记忆

客观事实，可验证。

例如：
- 用户会 Go
- 用户在上海
- 用户喜欢咖啡

**自动归类规则：** 包含"会"、"在"、"是"等判断词的陈述句。

------

### L2：偏好记忆

主观偏好，价值判断。

例如：
- 喜欢简洁架构
- 不喜欢复杂框架
- 偏爱稳定的技术

**自动归类规则：** 包含"喜欢"、"不喜欢"、"偏爱"、"讨厌"等偏好词的句子。

------

### L3：经历记忆

共同经历的事件。

例如：
- 第一次讨论 Agent
- 一起完成了学习计划
- 完成了项目第一阶段

**自动归类规则：** 包含时间词或事件词的句子。

------

### L4：关系记忆

体现双方协作变化的里程碑。

例如：
- 第一次完成项目
- 一起庆祝学习成果
- 建立信任的时刻

**自动归类规则：** 由 Reflection Agent 生成或用户明确标记。

------

### L5：价值观记忆

用户的核心价值观和信念。

例如：
- 长期主义
- 追求本质
- 喜欢掌控完整系统

**自动归类规则：** 包含"认为"、"相信"、"坚持"、"原则"等价值观词，或由 Reflection Agent 提炼。

------

## 6.3 记忆重叠处理

当一句话同时涉及多层记忆时：

1. 拆分为多条记忆分别存储
2. 每条记忆打上对应的层级标签
3. 保留原始对话作为引用

# 七、主动机制

## 原则

少而精。

------

## 触发条件

**仅在用户主动打开应用时**才可能触发主动内容。

绝不后台推送、绝不弹窗打扰。

------

## 主动内容类型

### 一级：观察

仅观察，不打扰。

在"伙伴"页面以卡片形式展示。

------

### 二级：轻提醒

例如："最近似乎很久没提 Agent 项目了。"

以聊天消息形式展示，由助手主动发起。

------

### 三级：重要节点

例如：项目完成、长期变化、学习里程碑。

以聊天消息形式展示。

------

## 频率控制

| 类型 | 每日上限 | 每周上限 |
|------|----------|----------|
| 二级提醒 | 1 条 | 3 条 |
| 三级节点 | 0 条 | 2 条 |

# 十二、错误处理与安全

## 大模型输出错误

- 胡编乱造：明确标识"这是推测"
- 记忆错误：用户可在记忆页面编辑修正
- 不当言论：拒绝输出，提供替代回应

------

## 用户纠正机制

- 直接否定："不对，我不是这个意思"
- 编辑记忆：在记忆页面修改
- 明确指示："请忘记这条""正确的应该是..."

------

## 隐私保护

- 所有数据存储于用户本地
- 不进行云同步
- 不收集用户数据用于训练
- API Key 加密存储

# 十六、用户故事

## US-001：首次使用引导

**用户：** 第一次打开应用的开发者

**场景：** 用户下载并首次启动 Along

**流程：**

| 步骤 | 用户操作 | 系统响应 | 备注 |
|------|----------|----------|------|
| 1 | 双击打开应用 | 显示欢迎页面 | 淡入动画 |
| 2 | 点击"开始" | 询问用户基本信息 | "你好，我是 Along。可以告诉我你的名字吗？" |
| 3 | 输入名字 | 确认并继续 | "好的，[名字]。已记录。" |
| 4 | 引导设置 API Key | 跳转到设置页面 | "需要配置 AI 服务才能正常工作。" |
| 5 | 完成配置 | 返回首页，显示欢迎消息 | "配置完成。现在可以开始了，有什么需要帮忙的吗？" |

**验收标准：**
- 首次启动必现引导流程
- 引导流程可跳过
- API Key 配置后立即生效

---

## US-002：情绪低落时的支持

**用户：** 加班到深夜，遇到技术难题的开发者

**场景：** 用户连续三天加班，Bug 始终无法解决，情绪崩溃

**流程：**

| 步骤 | 用户操作 | 系统响应 | 备注 |
|------|----------|----------|------|
| 1 | 打开应用，输入"我好难啊" | Emotion Agent 识别情绪 | 情绪标签：关注 |
| 2 | 等待回复 | Along 先理解状态，不直接给解决方案 | "先说说情况，我理解一下。" |
| 3 | 用户继续倾诉 | Along 倾听，偶尔回应 | "嗯，我在听。" |
| 4 | 用户情绪平复后 | Along 询问是否需要帮助解决问题 | "现在感觉好点了吗？如果需要，我们可以一起看看那个问题。" |

**验收标准：**
- 识别到负面情绪时，先理解后解决
- 不打断用户倾诉
- 情绪标签存储到记忆系统

## US-003：学习规划与跟踪

**用户：** 想要学习 Go 语言的开发者

**场景：** 用户决定学习 Go 语言，但不知道从何入手

**流程：**

| 4 | 输入"我今天不想学了" | Emotion Agent + Planner Agent 协作 | 先理解原因，再调整计划 |

**验收标准：**
- 学习计划可保存到目标表
- 进度自动跟踪
- 计划可调整
